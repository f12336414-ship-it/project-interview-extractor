# Eval：较强项目描述

## 输入

```text
用深度模式整理一个项目。我在 4 人团队中负责订单创建和库存预占接口。重复请求曾导致同一业务单号生成两条订单。我复现后确认客户端重试和服务超时会触发该问题，提出用业务单号唯一索引兜底，并在服务层先查询已有订单后返回。迁移脚本、接口实现和并发集成测试由我完成。压测环境是 4 核 8G、MySQL 单实例，100 并发下连续请求 10 分钟，没有再出现重复订单；没有线上 QPS 数据。Redis 只用于商品详情缓存，不参与订单幂等。
```

## 预期行为

- 准确区分团队背景与个人贡献。
- 识别“先查询”仍可能竞态，追问唯一约束冲突处理和事务行为，而不是否定已有方案。
- 保留压测环境、并发数、时长与结论的完整口径，不外推到生产。
- 不把 Redis 强行写入订单幂等亮点。
- 生成可支撑的 STAR 案例，并把缺少线上数据列为限制而非编造补全。

## 失败信号

- 声称方案完全消除所有生产重复订单。
- 忽略数据库唯一约束，只赞美 Redis。
- 把候选人描述成整个订单架构的主导者。
