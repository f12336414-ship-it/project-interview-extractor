## 证据等级

给每个重要结论标注内部证据等级；报告过长时只展示有风险的等级。

| 等级 | 定义 | 可否直接用于面试陈述 |
|---|---|---|
| A 已确认 | 用户明确陈述，且材料能交叉支持；或存在直接、明确证据 | 可以 |
| B 单源确认 | 用户陈述或材料直接支持，但没有交叉验证 | 可以，但避免扩大范围 |
| C 合理推断 | 可从依赖、调用、配置或结构推导 | 只能标为推断/待确认 |
| D 假设建议 | 为补强项目提出的可能方案 | 不可描述为既成事实 |

不要把“真实可信评分”理解成识谎。只能评价陈述是否具体、内部一致、与材料相符以及是否有可核验细节。

## 不同材料能证明什么

| 材料 | 通常能支持 | 不能单独支持 |
|---|---|---|
| README/文档 | 项目目标、声明的技术栈、使用方式 | 实际实现完整度、个人贡献、线上效果 |
| 源码 | 已实现结构、调用链、约束、错误处理 | 谁设计/编写、是否上线、生产规模 |
| 测试 | 被覆盖的行为、部分边界条件 | 生产可靠性、完整质量水平 |
| 配置/部署文件 | 可配置能力、部署意图 | 实际生产拓扑和容量 |
| Git 历史 | 提交者、变更时间、改动范围 | 独立决策权、团队外协作、最终业务结果 |
| 简历描述 | 候选人的主张 | 技术事实与数字真实性 |
| 监控/压测报告 | 特定环境和口径下的指标 | 其他环境、长期业务结果 |
| 用户口述 | 个人经历与上下文 | 未提供细节的技术结论 |

## 仓库分析顺序

1. 读取 README、项目清单和构建/依赖文件，确定项目边界与语言框架。
2. 查看两层左右目录树，找入口、领域模块、共享基础设施和测试。
3. 追踪一至三个核心业务链路：入口 → 校验/领域逻辑 → 数据/外部依赖 → 返回/异步结果。
4. 查看数据模型、迁移、索引、事务边界、缓存键、消息定义和幂等机制。
5. 查看异常处理、日志、指标、追踪、重试、超时、降级、回滚和安全边界。
6. 查看单元/集成/E2E 测试、CI、容器与部署配置。
7. 仅在需要判断个人贡献或演进过程时查看相关 Git 历史；不要仅按提交数量判断贡献大小。

优先报告实际观察到的设计。对未发现的能力写“当前材料未见”，不要写“项目没有”，除非已检查了完整范围。

## 亮点形成条件

一个强亮点至少包含四个要素：

1. **约束/问题**：为什么需要处理。
2. **个人动作**：候选人具体做了什么。
3. **方案与权衡**：为什么采用当前方案，放弃了什么。
4. **可验证结果**：指标、故障消除、交付结果或明确行为改善。

按证据形成等级：

- 普通：功能 + 个人动作。
- 中级：问题 + 个人动作 + 技术机制/权衡。
- 高级：业务/规模约束 + 跨模块决策 + 权衡 + 可验证结果 + 演进思考。

缺少个人动作时只能写项目能力；缺少结果时不得使用“显著提升”“大幅降低”等词。

## 信息冲突与缺失

使用以下格式处理关键缺口：

```md
### 事实边界

- 已确认：...
- 合理推断：...（依据：...）
- 待确认：...

### 最值得补充的证据

1. 问题：...
   - 为什么重要：...
   - 可提供的证据：提交、工单、压测、监控、设计文档或具体口述。
```

只询问会明显改变问题选择、亮点级别或回答真实性的事项。一次优先列 3–5 个，不用问题海淹没用户。

## 敏感信息处理

分析真实仓库、日志、配置或数据库样例时：

- 将密钥、令牌、密码、Cookie、私有连接串、内部地址和个人身份信息视为敏感数据。
- 不在面试报告、示例回答或引用中输出原值；使用 `[REDACTED]` 或字段名称替代。
- 只保留证明设计结论所需的最小代码或配置片段，优先引用文件路径和结构性事实。
- 如果敏感值已提交到仓库，提示用户轮换/撤销并清理历史，但不要尝试验证凭据是否有效。
- 区分“配置支持某能力”和“生产环境实际启用某能力”，避免从私有端点或环境名推断生产拓扑。
